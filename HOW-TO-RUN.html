How to Run AccessWidget

Complete setup guide — static pages, API server, and database

🌐
Static Pages
demo.html, admin/index.html, saas/dashboard/ — plain HTML, open directly in any browser.
No server needed
⚙️
API Server
Node.js + Express backend at saas/api/. Handles license validation, auth, analytics.
Node.js required
🗄️
SQLite Database
File-based database at saas/api/data/accesswidget.db. Auto-created on first run. No external server.
Auto-created
1
Open the Static Pages (no setup needed)

Open the demo page

Double-click any HTML file to open it in your browser. The widget works instantly — no server required for the front-end.

# Windows Explorer — just double-click:
demo.html
admin\index.html
saas\dashboard\login.html
saas\dashboard\index.html
embed-guide.html
✅ That's it for the front-end All pages open without any server. The widget, admin panel, and customer dashboard all work in the browser.

Optional: use a local HTTP server for better experience

Some browsers block file:// URLs from making fetch requests. If the widget's license check fails locally, serve the files over HTTP:

# Option A — Python (built-in, no install)
$ cd "c:\Users\admin\Downloads\Accessability tool"
$ python -m http.server 8080
# Then open: http://localhost:8080/demo.html
# Option B — Node.js (install once)
$ npm install -g serve
$ serve "c:\Users\admin\Downloads\Accessability tool" -p 8080
2
Install Prerequisites (one time only)

Install Node.js

The API server requires Node.js v18 or higher. Download from nodejs.org and run the installer.

# Verify installation after downloading:
$ node --version # should print v18.x.x or higher
$ npm --version # should print 9.x.x or higher
ℹ Download Node.js Visit https://nodejs.org → Download the LTS version → Run the .msi installer → Restart your terminal.
3
Start the API Server

Install dependencies

Open a terminal (Command Prompt or PowerShell) and run:

$ cd "c:\Users\admin\Downloads\Accessability tool\saas\api"
$ npm install

This installs: express, better-sqlite3, cors, helmet, uuid, express-rate-limit

Start the server

# Production (keeps running until you close the terminal):
$ npm start
# Development (auto-restarts when you edit server.js):
$ npm run dev
✅ Server is running when you see: URL : http://localhost:3001
DB : SQLite → saas/api/data/accesswidget.db

Test the API

Open your browser or use curl to verify:

# Health check — should return {"status":"ok"}
$ curl http://localhost:3001/health
# Test license validation (demo key):
$ curl -X POST http://localhost:3001/api/v1/validate \
     -H "Content-Type: application/json" \
     -d "{\"key\":\"AW-PRO1-X2Y3-Z4A5-B6C7\",\"domain\":\"mycompany.com\"}"

Database — What's Stored & Where

SQLite — file-based, no external server

The database is a single file that is created automatically the first time you run the server. All customer data persists between server restarts.

# Database file location:
saas\api\data\accesswidget.db
✅ Data survives server restarts Unlike the old in-memory version, all customers, licenses, and analytics are now saved permanently to disk.

⚠ Production note: hash your passwords Currently passwords are stored as plain text for demo simplicity. Before going live, install bcrypt (npm install bcrypt) and hash passwords on register and compare on login. Never store plain passwords in production.

View the database (optional)

To browse the SQLite database visually, download DB Browser for SQLite (free) and open saas/api/data/accesswidget.db.

# Or query it from the terminal:
$ sqlite3 saas\api\data\accesswidget.db
sqlite> SELECT email, plan, api_key FROM customers;
sqlite> SELECT key, domain, status FROM licenses;
sqlite> SELECT COUNT(*) FROM analytics WHERE type='activation';
